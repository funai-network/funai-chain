# V6 批日志重放验证 — 单机测试总结

**日期**：2026-04-21  
**受众**：项目成员、非技术协作者、未读详细报告的开发者  
**配套详细报告**：[`report.md`](report.md)

---

## 一句话结论

**V6 协议的核心假设"Worker 怎么跑，Verifier 就能一模一样重放一遍"在单台 GPU 上已经实证成立。** 26 个测试全部通过、零漂移，下一关是换台不同型号的 GPU 再测一次，过了就可以开始改主链代码了。

## 背景：为什么要做这件事

上周二（2026-04-20）发现 FunAI 主网一个硬伤：

- 当前验证方案（V5.2）假定 Worker 用"批推理"（batch inference，一张卡同时跑多个任务）拿到的 AI 内部打分，Verifier 用"单条推理"能复现出来
- 真机实测下来这两个打分**差 2.3%**，挑出的词从第 1 位就不一样了
- 也就是说 Worker 和 Verifier 对同一个输入会给出两套答案，**验证根本不成立**

KT 随后提出 **V6 方案**替代：Worker 除了输出答案，还要记录一份"batch 日志"——每一步哪些任务在同一批里、顺序是什么；Verifier 按日志**精确重放** Worker 的跑法。结果一字不差 = Worker 没作弊；只要有任何偏差，作弊即可判定。

方案的技术核心是两个假设：

- **A1**：有办法让推理引擎按日志精确重放 batch
- **A2**：同样的 batch 同样的内容，换台 GPU 也能得到完全一样的结果（否则不同 Verifier 用不同 GPU 就乱了）

**本次测试验证 A1 在单机上成立**。A2 留给下一轮跨机测试。

## 怎么测的

分 4 档逐步加压，每档过了再上下一档。哪档崩了能精确定位问题出在哪一步。

| 档 | 测什么 | 回答什么核心问题 |
|---|---|---|
| Phase 1a | 4 个任务同时跑，每次都挑最优词（`temperature=0`） | 最基本：Worker 和 Verifier 的内部评分能不能**逐位相等** |
| Phase 1c.1 | 4 个任务，有些提前结束离开批次 | 动态 batch：任务"离开"时 Verifier 还能跟上吗 |
| Phase 1c.2 | 4 个任务，有些中途才加入批次 | 动态 batch：任务"加入"时 Verifier 还能跟上吗 |
| Phase 1b | 同上但 `temperature=0.7`（有随机性的采样） | 加上随机采样（通过 ChaCha20 加密算法确定性化），Worker 和 Verifier 能不能都挑出同一个词 |
| Phase 1d | 把每个维度都拉大：12 任务、300 字长提示、30 步生成、6 任务 4 次切换；另加一个"引擎版本不匹配能不能拒绝"的校验 | 规模变大会不会漂移？协议防护到不到位 |

**测试机器**：阿里云华东 1 区杭州可用区 K，1 台 A10 GPU（24 GB 显存），spot 实例。  
**模型**：Qwen2.5 官方中文模型（0.5B 流水线快，3B 作为正式基准）。  
**推理引擎**：HuggingFace transformers（PoC 级，不是生产级，见后文"还没证明的事"）。

## 结果

严格指标：Worker 和 Verifier 的评分向量**逐位完全相等**（`max_abs_err == 0.0`，不容许一个比特的差别）。

| 场景 | Qwen2.5-0.5B（5 亿参数，快速验证流水线） | Qwen2.5-3B-Instruct（30 亿参数，正式基准） |
|---|---|---|
| Phase 1a 固定 batch | ✅ 6/6 | ✅ 6/6 |
| Phase 1c.1 任务离开 | ✅ 3/3 | ✅ 3/3 |
| Phase 1c.2 任务加入 | ✅ 3/3 | ✅ 3/3 |
| Phase 1b ChaCha20 采样 | ✅ 9/9 | ✅ 9/9 |
| Phase 1d 规模压测 | ✅ 5/5 | ✅ 5/5 |
| **合计** | **26/26 全过**（3 分 19 秒） | **26/26 全过**（11 分 11 秒） |

**没有一次漂移**。约 400 次 Worker–Verifier 评分比对 + 所有 ChaCha20 采样结果比对，零错配。

### 这个结果意味着什么

1. **V6 的地基立住了**：Worker 能产出足够详细的日志让 Verifier 精确重放——KT 提案的第一关键假设 A1 现在有代码 + 数据背书，不再是空话。
2. **V5.2 里的两个老问题顺带解决了**：
   - 老方案只能可靠验证 **7%** 的有温度任务（剩余 93% 因为 batch 漂移而采样验证不可靠）→ V6 方案下 **100% 都能精确验证**
   - 老方案防不了"采样操纵攻击"（AI 回答里偷偷塞广告、内容诱导、敏感词过滤）→ V6 下这类攻击从 "93% 检测不到" 变成 "100% 逐位抓住"。**这是 V6 相对 V5.2 最大的安全增益。**
3. **下一关明确**：单机 A1 过了，下一关是跨硬件 A2（不同型号 GPU 能否一致），已在计划中。

## 还没证明的事（诚实地讲）

V6 是协议级大动作，本次只走完了"单机 A1"这一关。剩下的：

| 事项 | 目前状态 | 关键性 |
|---|---|---|
| **跨硬件一致性**：A10 生成的日志换到 L20、4090 或 A100 上重放是否还一致 | 工程师私下测过一次说 OK，**但报告没入库**；下一步用阿里云换台不同型号机器再跑一遍 | ⭐⭐⭐ 核心 —— 这关挂了，V6 方案直接死，得回退到备选（Option B，代价更大） |
| **跨推理引擎**：Worker 跑 TGI、Verifier 跑 vLLM 能否一致 | 没测。KT 的方案是**"不跨引擎"**——每个模型注册时锁引擎 + 版本，钉死就行 | ⭐⭐ 协议层用"绑定"回避，这个方案可接受；PoC 里已实现元数据校验 |
| **真实生产引擎**：PoC 用的是学术代码（HuggingFace transformers），比生产级 vLLM / TGI 慢 10-20 倍。主网要移植到快引擎 | 预估工程量 2-3 个月；等跨硬件过了再启动 | ⭐⭐ 跨硬件之后的大工程 |
| **日志伪造防护**：Worker 不交日志已经能抓（判整批 FAIL），但交"一份自洽的伪造日志"这个攻击路径还没防 | V6 设计文档里没写，需要补设计 | ⭐⭐ 主网前必须补 |
| **Verifier 成本放大**：Verifier 要重放整个 batch 才能验一个任务——总成本约是单任务推理的 **几十倍**。现有 85%/12%/3% 的费率分配可能不够养 Verifier | 还没算经济账 | ⭐⭐ 经济模型待校对 |

## 附：测试档内容速览（给想了解细节的开发者）

### Phase 1a — 固定批

所有 4 个任务一起开始、一起跑 10 步生成、每步都挑最优词（无随机性）。Worker 用 HuggingFace 标准的 prefill+decode 流程跑一遍、保留所有内部打分；Verifier 重跑一遍；逐位对比。

### Phase 1c.1 — 任务离开

任务 1 跑 4 步就走、任务 2 跑 6 步、任务 3 跑 8 步、任务 4 跑全 10 步。每一步的"当前批次成员"动态变化。Worker 把每步的成员列表记进日志，Verifier 按日志重放。

### Phase 1c.2 — 任务加入 + 离开（最重要）

任务 1 从步 0 跑到步 4、任务 2 跑 0-7、任务 3 从步 2 才加入跑到步 8、任务 4 从步 4 才加入跑到步 9。**批次组成在多个时刻都发生变化**，这是 V6 最"真实"的场景——生产环境就是这个样子。Worker 和 Verifier 必须在每个组成变化的时刻都保持一致。

### Phase 1b — ChaCha20 确定性采样

启用随机性（`temperature=0.7`），但用 ChaCha20 加密算法把"随机"变成"确定性"——给定同样的种子和内部状态，每次挑到的词都一样。Worker 和 Verifier 只要都按 V5.2 §9.3 的规定实现采样（fp32 精度、特定的累加顺序、ChaCha20 种子推导方式），就能挑出同一个词。

### Phase 1d — 压力测试 + 协议校验

五个场景各自独立：
1. **12 任务同时跑**——批大小变大会不会出现新问题
2. **300 字长提示词**——长输入会不会碰上数值稳定性边界
3. **30 步生成**——长时间生成会不会误差累积
4. **6 任务 4 次切换**——更密集的加入/离开转换
5. **引擎版本不匹配应拒**——如果 Worker 跑 transformers 4.57.6、Verifier 跑 4.50.0，Replayer 是否拒绝（保护 KT v2 §2.5 "Worker 和 Verifier 必须用声明的引擎和版本"这一协议规定）
